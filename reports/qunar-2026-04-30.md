# com.Qunar (去哪儿) — anti-Frida / anti-debug profile

**Date:** 2026-04-30 (revised: trace v5 with extended syscall set)
**Device:** Xiaomi alioth (Redmi K40), kernel `4.19.325-cip128-st12-perf-g79a70a234c00`
**App:** 去哪儿旅行 on alioth (uid `10166`, two-process: main + `:qutui`)
**Trace duration:** 70 s (cold start + simulated UI taps + scrolls), attach-before-launch, uid filter `-u 10166`
**Syscalls hooked:** `openat, connect, readlinkat, faccessat, newfstatat, statx, prctl, ptrace, mprotect, pread64, socket, bind, clone, kill, getsockname`
**Sample size:** 104 428 `sys_enter` events, 227+ distinct paths

> **execve hook attempt note.** We tried to add `execve,execveat` to the
> syscall list (the natural follow-up to the previous Qunar pass).
> stackplz silently dropped these — even when triggered manually from a
> root shell with `-u 0` filter, zero `execve` events were produced.
> The BPF-side filter or arg-parser appears to swallow them.  Tracking
> this as an upstream stackplz issue, separate from miku-shield.

## TL;DR

Qunar's detection model is **opposite to TB's** in two big ways and one
subtle one we missed last pass:

1. **Qunar shells out heavily** via `Runtime.exec` to system utilities
   (`getprop`, `sh`, `cat`, `ps`, `ls`, `pm`, `ip`, `which`, `stat`,
   `grep`, `cmd`).  In this 70-s trace there were **626 `clone()` calls**
   (184 from `sh` threads alone) and tens of thousands of subprocess
   syscalls.
2. **Qunar uses sockets for sock-diag-style enumeration** — 9 003
   IPv4 SOCK_STREAM sockets created, 5 netlink sockets created and
   `bind()`-ed to all-groups `nl_groups=0xFFFFFFFF`.  This is the
   `NETLINK_SOCK_DIAG` pattern for enumerating *all* TCP sockets on
   the device without ever opening `/proc/net/tcp` — perfect for
   spotting `frida-server`'s listener.
3. **Tencent QAPM** does its background polling — 278 `/proc/self/exe`
   reads, 46 `/proc/self/maps` scans, plus 222 task-status enumeration
   reads after the user interacts with the UI.

## Socket-based detection (new in v5)

### Socket creation rate

| domain    | type                                                    | count |
|-----------|---------------------------------------------------------|------:|
| AF_INET   | SOCK_STREAM                                             | 9 003 |
| AF_UNIX   | SOCK_DGRAM \| SOCK_CLOEXEC \| SOCK_NONBLOCK             | 21    |
| AF_INET6  | SOCK_DGRAM \| SOCK_CLOEXEC                              | 14    |
| AF_UNIX   | SOCK_STREAM \| SOCK_CLOEXEC                             | 14    |
| AF_INET   | SOCK_DGRAM \| SOCK_CLOEXEC                              | 12    |
| AF_NETLINK| SOCK_STREAM \| SOCK_DGRAM \| SOCK_RAW \| SOCK_CLOEXEC   | **5** |
| AF_UNIX   | SOCK_STREAM                                             | 2     |

9 003 IPv4 TCP sockets in 70 s — these are not all connected (the
`connect()` count is much lower), so most exist only briefly.  Possible
purposes: probing local ports without actually completing handshakes,
or `getsockname()` / `SO_*` queries that Qunar uses to fingerprint the
network namespace.

### `AF_NETLINK` bind to all groups

```
socket(AF_NETLINK, SOCK_RAW|SOCK_CLOEXEC, ?)   × 5
bind(sockfd, &(struct sockaddr_nl){
  .nl_family = AF_NETLINK,
  .nl_pid    = 0,
  .nl_groups = 0xFFFFFFFF        ← subscribe to ALL multicast groups
}, 12)                                          × 3
```

This is the **`NETLINK_SOCK_DIAG`** pattern: subscribe to socket-diag
events to enumerate every TCP/UDP/UNIX socket on the device, without
ever opening `/proc/net/tcp`.  A bypass that targets `/proc/net/tcp`
reads (which our existing `proc-net-tcp-scan` detector covers) will
miss this entirely.

> **Mitigation implication for traceMiku.** The frida-server abstract
> UNIX socket (or any helper-process socket) IS visible via
> `NETLINK_SOCK_DIAG` to anyone who binds to the right groups.
> Re-naming the socket does not help — the socket name appears in the
> SOCK_DIAG response.  The only way to hide is to not have a listening
> socket in the same network namespace as the target app.

### `connect()` targets

| target | count |
|--------|------:|
| AF_UNIX `/dev/socket/dnsproxyd` (DNS) | ~14 |
| AF_UNIX `/dev/socket/logdw` (logd writer) | ~21 |
| AF_UNSPEC (binary payload) | ~3 000 |
| TCP / UDP IPv4 / IPv6 | **0** |

The "AF_UNSPEC" calls are `connect(sock, &(sockaddr){.sa_family = AF_UNSPEC}, ...)`
which the kernel interprets as "disconnect this socket".  Qunar appears
to use this as a bulk-cleanup pattern after bursts of socket creation.

## clone() bursts — when subprocess scans fire

```
t=  0– 5s    2 clones    ← startup
t=  5–10s   13 clones    ← splash + first detection sweep
t= 15–20s    1 clone
t= 25–30s   14 clones    ← second sweep
t= 35–40s    1 clone     ← idle
t= 55–60s   85 clones   ┐
t= 60–65s   68 clones   │  ← burst — Qunar woke up after our UI back-press
t= 65–70s  129 clones   ┘
                ────
total           626 clones
```

`sh` accounted for 184 of the 626 clones — every `sh -c "getprop X | grep Y"`
fork+exec pipeline shows up as a `sh` clone.  The big burst at 55–70 s
correlates with our UI-tap simulation: tapping causes Qunar to re-run
a chunk of detection logic.

kill(24778, SIGTERM)    ← Qunar reaping its own subprocess

Just normal `Runtime.exec` cleanup.  Not detection.

## Tencent QAPM polling (t = 54–66 s)

A burst of 278 `/proc/self/exe` `openat` calls in a 12-s window, all
from `QTask #N` threads.  This is QAPM's heartbeat:

```
54.339 ─┐
        │   278 reads of /proc/self/exe
        │   from QTask threads, ~24/s
66.447 ─┘
```

QAPM is **Tencent**'s app-performance monitoring SDK, used in many
Tencent products and licensed to non-Tencent apps that use Tencent
CDN/anti-cheat.  Seeing QAPM in 去哪儿 confirms Qunar uses Tencent's
anti-cheat suite.

## Comparison with TB

|                                       | TB         | Qunar     |
|---------------------------------------|-----------|-----------|
| **Detection style**                   | inline syscalls | shell-out + sock-diag |
| `Runtime.exec` subprocess count / 60 s| ~0        | ~10 000   |
| `clone()` count / 70 s                | ~0        | 626       |
| `ptrace(PTRACE_TRACEME)`              | 1 (DaemonThread-6) | 0 |
| `socket(AF_INET, SOCK_STREAM)` count  | 0         | 9 003     |
| `socket(AF_NETLINK, ...)` count       | 0         | 5         |
| `bind(AF_NETLINK, all_groups)`        | 0         | 3         |
| `/proc/self/maps` reads               | 2         | 46 (QAPM) |
| `/proc/self/exe` reads                | 3         | 278 (QAPM)|
| Thread-comm / status enumeration      | yes (heavy at startup) | yes (after UI tap, 222 events) |
| `/proc/net/unix` scan                 | 10        | 0         |
| MAC address read                      | 4 NICs × 5 = 20 | (via `ip` subprocess + netlink, not openat) |
| Hardest-to-bypass mechanism           | private-cache `readdir` | shell-out + NETLINK_SOCK_DIAG |

**Different threat models entirely**:
- **TB** assumes you can bypass any single in-process check, and stacks many
  short fast checks across all major detection axes (path, thread, maps, status,
  cmdline, exe).
- **Qunar** assumes you can hook libc / direct syscalls but you cannot
  perfectly emulate `Runtime.exec` returning expected stdout, AND you
  cannot hide listening sockets from `NETLINK_SOCK_DIAG`.
  The shell-out is invisible to syscall hooks because the signal is in
  subprocess argv, and the netlink path bypasses every `/proc/net/tcp`
  hook.

## What miku-shield's path-based detector still misses

1. **Subprocess argv** — what *exactly* is being passed to `getprop`,
   `cat`, `pm`, `ps`?  Needs `execve`-with-argv hooking, currently
   broken in stackplz.
2. **Netlink message bodies** — what diag families is Qunar querying?
   Needs `sendmsg`/`recvmsg` hooking with sockaddr_nl decode.
3. **Socket creation rate** — 9 003 IPv4 sockets is anomalous on its own.
   Our YAML schema is path-based; rate-based rules would need a new
   detector kind.

## Open follow-ups

- **Fix execve hooking in stackplz** — the syscall is in the config and
  the whitelist, but events never fire.  Likely a BPF-side filter or
  string_array arg-parser bug.  Without this, the most informative
  signal for Qunar (subprocess argv) is invisible.
- **Add `sendmsg`/`recvmsg` to the trace** to capture netlink message
  bodies — would reveal which sock-diag family Qunar queries.
- **Add a rate-based detector kind** ("9000+ IPv4 SOCK_STREAM creations
  in <60 s from a single uid is anomalous") — current schema is
  path-glob-only.
- **`:qutui` subprocess role** — runs in same uid 10166 so it's already
  in the trace, but its specific role (auth? push?) was not isolated
  here.
- **Compare with QQ when APK becomes available** — Tencent QAPM should
  show up there too with much more activity than we see in Qunar (which
  only licenses parts of QAPM).
