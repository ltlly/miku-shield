# com.Qunar (去哪儿) — anti-Frida / anti-debug profile

**Date:** 2026-04-30
**Device:** Xiaomi alioth (Redmi K40), kernel `4.19.325-cip128-st12-perf-g79a70a234c00`
**App:** 去哪儿旅行 on alioth (uid `10166`, two-process: main + `:qutui`)
**Trace duration:** 60 s (cold start + simulated UI taps), attach-before-launch, uid filter `-u 10166`
**Syscalls hooked:** `openat, connect, readlinkat, faccessat, newfstatat, statx, prctl, ptrace, mprotect, pread64`
**Sample size:** 29 163 `sys_enter` events, 227 distinct paths

## TL;DR

Qunar's detection model is **opposite to TB's**: instead of doing in-process syscalls, Qunar **shells out** via `Runtime.exec()` to Android system utilities (`getprop`, `sh`, `cat`, `ps`, `ls`, `pm`, `ip`, `which`, `stat`) and parses their stdout.  In a 60-s trace there were 13 111 spawned `getprop` events, 4 200 `sh`, 2 544 `cat`, 676 `ps`, 643 `ls`, 642 `grep`, 602 `pm`, 290 `ip` — versus only **628 events from threads with `com.Qunar` `comm`**.  The main process itself does relatively little detection work directly.

| signal class                             | count | severity |
|------------------------------------------|------:|----------|
| `proc-self-exe-check` (QAPM polling)     | 80    | LOW      |
| `proc-self-maps-scan`                    | 12    | MED      |
| `proc-self-cmdline`                      | 8     | LOW      |
| `tracerpid-status` (`/proc/self/status`) | 1     | MED      |
| `cpu-info` (`/proc/cpuinfo`)             | 1     | LOW      |
| `ptrace`                                 | 0     | —        |

**Total via miku-shield analyze:** 102 detections, 5 distinct rules, MED ×13 / LOW ×89.

This number understates the real story — most of Qunar's detection signal is **invisible to a path-glob detector** because it happens through subprocess argv (which we don't trace).  The 80 `/proc/self/exe` reads at t = 54–57 s are the **Tencent QAPM** monitoring framework polling its own binary; it's a reliable, easy-to-spot fingerprint that this app integrates QAPM.

`cat` did read these once each:


## Tencent QAPM polling (t = 54–57 s)

A burst of 80 `/proc/self/exe` `openat` calls in a 3-s window, all from `QTask #N` threads.  This is QAPM's heartbeat:

```
54.339 ─┐
54.349  │   80 reads of /proc/self/exe
...     │   from QTask threads, ~12/s
56.581 ─┘
```

QAPM is **Tencent**'s app-performance monitoring SDK, used in many Tencent products and licensed to non-Tencent apps that use Tencent CDN/anti-cheat.  Seeing QAPM in 去哪儿 confirms Qunar uses Tencent's anti-cheat suite.

## Comparison with TB

|                                     | TB         | Qunar    |
|-------------------------------------|-----------|----------|
| **Detection style**                 | inline syscalls | shell-out via Runtime.exec |
| `Runtime.exec` subprocess count / 30 s | ~0       | ~10 000  |
| `ptrace(PTRACE_TRACEME)`            | 1 (DaemonThread-6) | 0 |
| `/proc/self/maps` reads             | 2         | 12 (QAPM)|
| `/proc/self/exe` reads              | 3         | 80 (QAPM)|
| Thread-comm / status enumeration    | yes (heavy)| no       |
| `/proc/net/unix` scan (frida-server detect) | 10 | 0 |
| MAC address read                    | 4 NICs × 5 = 20 | (via `ip` subprocess, not openat) |
| Hardest-to-bypass mechanism         | private-cache `readdir` | shell-out — invisible to syscall hooks |

**Different threat models entirely**:
- **TB** assumes you can bypass any single in-process check, and stacks many
  short fast checks across all major detection axes (path, thread, maps, status,
  cmdline, exe).
- **Qunar** assumes you can hook libc / direct syscalls but you cannot
  perfectly emulate `Runtime.exec` returning expected stdout.  By piping
  `getprop ro.debuggable | grep ...` through real `sh` + `getprop`,
  Qunar gets evidence that's hard to fake without intercepting subprocess
  pipes.

## Why miku-shield's path-based detector misses most of Qunar

Our detector library matches `path_glob` on `openat()` events.  But Qunar's
detection is in the **arguments to spawned commands**, e.g.
`Runtime.exec("getprop ro.debuggable")`.  We never see "ro.debuggable" —
we only see `getprop` opening `/dev/__properties__/u:object_r:default_prop:s0`,
which is the same path it'd open for any property fetch.

To detect Qunar-style anti-debug we'd need to also trace **`execve` and grab argv** in the BPF.  That's a future addition.

## Reproducer

```bash
# Same kernel + binary requirements as the TB reproducer.
adb shell "
  am force-stop com.Qunar
  killall -9 com.Qunar com.Qunar:qutui 2>/dev/null
  sleep 2
  cd /data/local/tmp && timeout 65 ./stackplz --btf -j -u 10166 \
    --syscall openat,connect,readlinkat,faccessat,newfstatat,statx,prctl,ptrace,mprotect,pread64 \
    > /tmp/qunar.jsonl &
  sleep 2
  am start -W -n com.Qunar/com.mqunar.splash.SplashActivity
  sleep 8
  input tap 540 1200 ; sleep 4
  input swipe 540 1500 540 500 ; sleep 4
  input keyevent KEYCODE_BACK ; sleep 4
  input tap 540 1200 ; sleep 30
"
adb pull /tmp/qunar.jsonl
miku-shield analyze --output pretty < qunar.jsonl
```
