# com.taobao.taobao — anti-Frida / anti-debug profile

**Date:** 2026-04-30
**Device:** Xiaomi alioth (Redmi K40), kernel `4.19.325-cip128-st12-perf-g79a70a234c00`
**App version:** Taobao on alioth (uid `10189`, app version embedded in `libsgmainso-6.8.260403.so`)
**Trace duration:** 30 s, attach-before-launch, uid filter `-u 10189`
**Syscalls hooked:** `openat, connect, readlinkat, faccessat, newfstatat, statx, prctl, ptrace, mprotect, pread64`
**Sample size:** 25 562 `sys_enter` events, 759 distinct paths

## TL;DR

TB's anti-detection layer kicks in **within the first 100 ms** of process start. The bulk of it is in-process — direct `openat()` / `prctl()` / `mprotect()` calls, no `Runtime.exec`. The single most concentrated burst is at **t = 0.15–0.18 s**: enumerate all threads, read every `task/<tid>/{comm,status,stat}`, scan `/proc/self/maps`, check `/proc/self/status` for TracerPid, validate `/proc/self/exe`. After that startup spike, TB settles into periodic background polling driven by `libsgmainso`'s `sgFile.lock` (~24 / s).

```

0.14s  /proc/version                           ← kernel fingerprint
0.15s  /proc/self/task                         ← list threads
0.15s  /proc/self/task/<tid>/comm × 5          ← thread name (find frida-agent threads)

0.16s  /proc/self/exe                          ← own binary path
0.17s  /proc/self/task/<tid>/stat × 23         ← every thread's state (find suspicious threads)

```

## Notable absences (in this 30-s window)

- ❌ no TCP `connect()` to `127.0.0.1:27042` / `27043` (Frida default ports)
- ❌ no `Runtime.exec` shell-out (very different from Qunar; see `qunar-2026-04-30.md`)
- ❌ no probe of `/data/local/tmp/frida-server*` (TB tries this in some
  builds but did not in this trace — possibly because it runs that probe
  even earlier, before our BPF attach window)

## Private-dir enumeration (corrected interpretation)

TB walks its own `/data/user/0/com.taobao.taobao/cache/` with `readdir()`
+ `openat(O_DIRECTORY|O_CLOEXEC)`. In this trace it iterated 14 entries:

```
.miku, WebView, font-family, instant-opt, metrickit, metrickit/upload,
nano_fonts, nano_fonts/nano_files, nano_fonts/nano_metadata,
uc_debug, webview_com.taobao.taobao, instant-opt, tbp-dump, video-cache
```

`.miku` and `uc_debug` are flagged HIGH not because TB hunts them by
name, but because **they should not exist** under TB's private cache —
TB's enumeration sweep catches any unexpected entry regardless of name.
The implication is concrete: traceMiku and similar tools must not drop
any residue in the target's app-private directory; renaming is not a
mitigation. See the comment block in `data/known_detectors.yaml`.

## Reproducer

```bash
# device must be running our patched 4.19 kernel (g79a70a234c00 or later)
# /data/local/tmp/stackplz must equal the miku-shield binary
adb shell "
  am force-stop com.taobao.taobao
  cd /data/local/tmp && timeout 35 ./stackplz --btf -j -u 10189 \
    --syscall openat,connect,readlinkat,faccessat,newfstatat,statx,prctl,ptrace,mprotect,pread64 \
    > /tmp/tb.jsonl &
  sleep 2
  am start -W -n com.taobao.taobao/com.taobao.tao.welcome.Welcome
  sleep 30
"
adb pull /tmp/tb.jsonl
miku-shield analyze --output pretty < tb.jsonl
```
